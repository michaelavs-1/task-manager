## 1. תמונה כוללת — מה זה "מערכת הנהלת חשבונות"

מערכת הנהלת חשבונות מלאה היא צרור של חמישה רבדים מצטברים:

**רובד 1 — תיעוד תנועות גולמי (Source of Truth):** חשבוניות מס, חשבוניות מס/קבלה, קבלות, חשבוניות ספקים, תלושי שכר, דפי בנק, פנקסי צ'קים. כל מסמך הוא ראיה ראשונית.

**רובד 2 — פקודות יומן (Journal Entries) בשיטה דו-צדדית:** כל מסמך מתורגם לזוג חיוב/זיכוי שמסכם תמיד לאפס. זה הלב של "דו-צדדי" — המשוואה החשבונאית `נכסים = התחייבויות + הון` חייבת תמיד לאזן.

**רובד 3 — ספר ראשי וחשבונות עזר (General Ledger + Subsidiary Ledgers):** כל פקודת יומן מצטברת ל-T-account של כל חשבון בתמ"ה (תרשים מספור החשבונות, Chart of Accounts). בנוסף, ספרי עזר נפרדים לחייבים (לקוחות), זכאים (ספקים), מלאי וקופה.

**רובד 4 — דוחות תקופתיים (Period-End Reporting):** מאזן בוחן, רווח והפסד, מאזן, תזרים מזומנים — כולם נגזרים אוטומטית מהספר הראשי בכל רגע נתון.

**רובד 5 — ציות ודיווח לרגולטור (Compliance & Filing):** דיווח מע"מ תקופתי, PCN874, ניכוי מס במקור (102 חודשי, 856 שנתי), טופס 126 (שכר), טופס 6111 (דוחות כספיים שנתיים), חשבוניות ישראל (SHAAM allocation numbers).

מערכת מודרנית מוסיפה גם: אוטומציה (קליטה אוטומטית של דפי בנק, OCR לקבלות, חשבוניות חוזרות), שיתוף עם רו"ח (advisor portal + audit trail), ויזואליזציה (KPIs, תזרים חזוי, תקציב מול ביצוע).

## 2. רגולציה ישראלית — הוראות ניהול ספרים

### 2.1 הבסיס החוקי

החוק המסגרת: **תקנות מס הכנסה (ניהול פנקסי חשבונות) התשל"ג–1973**, במיוחד **תקנה 36** (שיטת רישום) ו-**תקנה 36א** (דרישות לתוכנה ממוחשבת).

### 2.2 שיטה חד-צדדית מול דו-צדדית

- **חד-צדדית** (single-entry): רישום פשוט של תקבולים והוצאות — מותר לעוסקים זעירים ולחלק מהענפים בהתאם להוראות פרטניות.
- **דו-צדדית** (double-entry, דביט/קרדיט): חובה עבור:
  - כל החברות בע"מ (ללא תלות בגודל)
  - עוסקים שמחזורם השנתי מעל סף שנקבע בענף הספציפי (משתנה לפי ענף; ברוב הענפים סביב מיליון ש"ח)
  - מי שמחזיק מלאי
- **הטבה:** עוסק שמנהל ספרים בשיטה דו-צדדית מקבל הארכה בהגשת הדו"ח השנתי עד 31 במאי (לעומת 30 באפריל).

### 2.3 ספרים נדרשים

| ספר | תפקיד |
|---|---|
| יומן (Journal) | רישום כרונולוגי של כל פקודת יומן |
| ספר ראשי (General Ledger) | T-accounts לכל חשבון, מהווה בסיס למאזן בוחן ולדוחות |
| ספר עזר (Subsidiary Ledger) | פירוט פרטני לחייבים, זכאים, מלאי |
| ספר קופה (Cash Book) | תקבולים ותשלומים במזומן |
| ספר תקבולים/תשלומים (Receipts & Payments) | תזרים יומי |
| ספר מלאי (Inventory Book) | רישום מלאי בסוף שנה (חובה לעוסקים שיש להם מלאי) |

### 2.4 שמירת תיעוד

- מסמכי הנהלת חשבונות: **7 שנים**
- תיעוד מס: **10 שנים** מסוף שנת המס שבה הוגש הדו"ח
- מותר לשמור דיגיטלית, אבל חייב להיות אפשר לאחזר במהירות לביקורת
- קנס על פנקסים בלתי קבילים (מ-1.7.2024): 1% מחבות המע"מ, מינימום ₪359

### 2.5 קוד חשבונות אחיד (טופס 6111)

ישראל אינה מחייבת קוד חשבונות אחיד אבל **טופס 6111** (נספח לדו"ח השנתי) מחייב את הדו"חות הכספיים בקודי חשבון תקניים שקבעה רשות המסים. בפועל, רוב התוכנות מאמצות את המספור של 6111 כברירת מחדל — מה שהופך אותו לסטנדרט דה-פקטו.

מבנה כללי של תרשים חשבונות ישראלי טיפוסי:

| טווח | קטגוריה |
|---|---|
| 1000–1999 | נכסים (מזומנים, חייבים, מלאי, רכוש קבוע, פחת נצבר) |
| 2000–2999 | התחייבויות (ספקים, מע"מ, מס הכנסה, ביטוח לאומי, הלוואות) |
| 3000–3999 | הון (הון מניות, עודפים, רווח שוטף) |
| 4000–4999 | הכנסות (חייבות, פטורות, אחרות) |
| 5000–5999 | עלות המכר / עלויות ישירות |
| 6000–6999 | הוצאות תפעוליות (שכר, שכ"ד, פרסום, רו"ח, ביטוח) |
| 7000–7999 | הוצאות פיננסיות (ריבית, עמלות בנק, הפרשי שער) |
| 9000–9999 | חשבונות ניטרליים / סגירה |

## 3. תהליכים חשבונאיים מרכזיים

### 3.1 משוואת הזהות והכללים

המשוואה: **נכסים = התחייבויות + הון**

| סוג חשבון | יתרה רגילה | חיוב מעלה | זיכוי מעלה |
|---|---|---|---|
| נכסים | חובה | ✓ | — |
| התחייבויות | זכות | — | ✓ |
| הון | זכות | — | ✓ |
| הכנסות | זכות | — | ✓ |
| הוצאות | חובה | ✓ | — |

### 3.2 שש פקודות יומן יסודיות

**א. הוצאת חשבונית ללקוח (₪1,000 + 18% מע"מ = ₪1,180):**
```
חובה   1150  לקוח X (חייבים)            1,180
זכות         4000  הכנסות משירותים           1,000
זכות         2310  מע"מ עסקאות (לתשלום)        180
```

**ב. קבלת תשלום עם ניכוי מס במקור 5% (חשבונית ₪10,000):**
```
חובה   1010  בנק                         9,500
חובה   2550  מס מקור — לקיזוז ממס שנתי     500
זכות         1150  לקוח X                   10,000
```

**ג. רישום הוצאה עם מע"מ תשומות (ספק Y, ₪500 + ₪90 מע"מ):**
```
חובה   6400  משרדיות                       500
חובה   2320  מע"מ תשומות (לקיזוז)           90
זכות         2000  ספק Y                       590
```

**ד. תשלום שכר חודשי (ברוטו ₪12,000):**
```
חובה   6000  הוצאות שכר — דוד          12,000
זכות         1010  בנק (נטו לעובד)            8,830
זכות         2410  מס הכנסה לתשלום            1,440
זכות         2460  ביטוח לאומי — עובד            630
זכות         2470  ביטוח בריאות                  380
זכות         2480  פנסיה — עובד                  720
```
(ובנוסף פקודה נפרדת להפרשות מעסיק — ביטוח לאומי, פנסיה, השתלמות.)

**ה. רישום פחת חודשי (ציוד ₪50,000, פחת 10% שנתי):**
```
חובה   6900  הוצאות פחת                    417
זכות         1750  פחת נצבר — ציוד               417
```

**ו. סגירת השנה — סגירת הכנסות והוצאות לעודפים:**
```
חובה   4000  הכנסות (סגירה)            500,000
זכות         3210  סיכום רווח והפסד         500,000

חובה   3210  סיכום רווח והפסד          350,000
זכות         6XXX  כל ההוצאות               350,000

חובה   3210  סיכום רווח והפסד (רווח נטו) 150,000
זכות         3110  עודפים                  150,000
```

### 3.3 תהליך סגירה תקופתי (חודש/רבעון/שנה)

1. **רישום כל התנועות** עד היום האחרון של התקופה
2. **התאמת ספרי עזר** לספר הראשי (חייבים מול 1150, ספקים מול 2000, בנק מול 1010)
3. **הפקת מאזן בוחן ראשוני** (Trial Balance) — סך חובה חייב להשתוות לסך זכות
4. **פקודות התאמה** (Adjusting Entries):
   - הפרשה לחובות מסופקים
   - הערכת מלאי מחדש (סוף שנה)
   - פחת
   - הוצאות והכנסות נדחות / שטרם התקבלו
5. **מאזן בוחן מתואם**
6. **הפקת דוחות כספיים** (רווח והפסד, מאזן, תזרים)
7. **בסוף שנה:** ספירת מלאי, סגירת חשבונות זמניים לעודפים, מאזן בוחן לאחר סגירה

### 3.4 פחת — שיעורים ישראליים

| נכס | שיעור שנתי |
|---|---|
| בניינים | 1.5%–6.5% |
| ציוד וחומרה | 6%–15% |
| רכב נוסעים (דלק/דיזל) | 15% |
| רכב היברידי | 25% |
| משאיות / רכב מסחרי | 20% |
| מחשבים וציוד IT | 20%–33% |
| רהיטים וציוד משרדי | 10%–13% |

**שיטות:** קו ישר (סטנדרטי בישראל ורוב הנכסים) או יתרה פוחתת (מאיץ פחת בשנים הראשונות).

### 3.5 התאמת בנק (Bank Reconciliation)

תהליך חודשי:
1. השוואת יתרת ספרים (1010) ליתרת דף הבנק
2. זיהוי הפרשי תזמון: הפקדות בדרך, צ'קים שטרם נפדו, עמלות בנק שטרם נרשמו, ריבית
3. רישום פקודות התאמה
4. יתרת ספרים מתואמת = יתרת בנק

### 3.6 דוחות גיול

**גיול חייבים (AR Aging):** טבלה של חשבוניות פתוחות לפי גיל (שוטף, 0–30, 31–60, 61–90, 90+). משמש להפרשה לחומ"ס ולמאמץ גבייה.

**גיול ספקים (AP Aging):** מקביל, לתכנון תזרים והעדפת ספקים.

## 4. דוחות ודיווחים חובה לרשות המסים

| דוח | טופס | תכיפות | פורמט | מקום הגשה |
|---|---|---|---|---|
| דיווח מע"מ תקופתי | 874 / 875 | חודשי / דו-חודשי | טקסט / שע"מ | secapp.taxes.gov.il |
| קובץ אחיד מע"מ | PCN874 | עם הדיווח | טקסט fixed-width | שע"מ |
| ניכוי מס חודשי | 102 | חודשי / דו-חודשי | טקסט / API | secapp.taxes.gov.il/nikpay |

### 4.1 דיווח מע"מ תקופתי (874)

- **עוסק חודשי:** מחזור > ₪1.5M שנתי, דיווח עד ה-23 לחודש שלאחריו
- **עוסק דו-חודשי:** מחזור < ₪1.5M, דיווח כל חודשיים עד ה-23
- **עוסק זעיר:** מחזור < ₪122,833 (2026) — פטור מדיווח
- שדות: עסקאות חייבות, עסקאות פטורות, מס עסקאות, מס תשומות על ציוד, מס תשומות אחרות, סיכום

### 4.2 קובץ אחיד PCN874

קובץ מובנה (fixed-width text) עם 4 סוגי רשומות:
- **A** — Header (מידע על העסק והתקופה)
- **B** — שורות חשבוניות מס שהוצאו
- **C** — שורות מס תשומות
- **D** — Summary (סיכום מצטבר)

חובה למחזור > ₪1.5M (או למי שבחר בדיווח מפורט). חייב להיות עקבי לחלוטין עם 874.

### 4.3 חשבוניות ישראל (SHAAM CTC)

תקנה שנכנסה לתוקף 5.5.2024 ומתפרסת בהדרגה:
- **מעל ₪25,000** (כיום): חובה לקבל "מספר הקצאה" מרשות המסים בזמן אמת לפני הוצאת חשבונית
- צפי: ירידת הסף ל-₪10,000 ואז ל-₪5,000 עד יוני 2026
- **API:** `secureapi.taxes.gov.il`, JSON, סביבת sandbox + production
- ללא מספר הקצאה — הלקוח לא יכול לקזז מע"מ תשומות
- חובה להטמיע הקצאה אוטומטית על כל חשבונית B2B מעל הסף

### 4.4 ניכוי מס במקור (102 ו-856)

- **102 חודשי:** דיווח כל ניכויי המס שבוצעו (מעובדים ומספקים) + תשלום עד ה-15 בחודש שלאחריו
- **856 שנתי:** ריכוז שנתי לכל ספק שניכינו ממנו מס — שם, ת"ז/ח.פ., ברוטו, ניכוי, נטו, אחוז ניכוי. עד 30 באפריל
- **תעודה לספק:** העסק חייב להפיק "תעודת ניכוי מס במקור" לכל ספק לסוף השנה

### 4.5 טופס 6111

נספח חובה לדו"ח השנתי לכל מי שמחזורו מעל ₪300K. מצריך:
- דו"ח רווח והפסד מבונה לפי קודי 6111
- מאזן מבונה לפי קודי 6111
- דו"ח התאמה למס (חישוב ההפרשים בין רווח חשבונאי לרווח לצורכי מס)

זו הסיבה שתוכנות חשבונאיות בישראל מאמצות את קודי 6111 כתרשים חשבונות ברירת מחדל.

### 4.6 קובץ אחיד BKMV

קובץ ZIP שמייצא את כל ספרי החשבונות (יומן, ספר ראשי, רשימות) בפורמט אחיד שכל תוכנה רשומה חייבת לתמוך בו. מבנה:
- `INI.TXT` — מידע על העסק והתקופה
- `BKMVDATA.TXT` — הנתונים בפועל
- `README.TXT` — מטא-דאטה

חובה ליכולת ייצוא BKMV כדי שתוכנה תוכר על ידי רשות המסים.

## 5. בנצ'מרק — מערכות ישראליות

### 5.1 חשבשבת (Hashavshevet) — הסטנדרט של רואי החשבון

- **שייכות:** Wizsoft, מ-1983
- **קהל יעד:** רואי חשבון, חברות בינוניות-גדולות
- **חוזקות:**
  - תקן הזהב להנה"ח כפולה בישראל; 50,000+ משתמשים היסטוריים
  - GL מלא, AP/AR, פרויקטים, רכוש קבוע עם פחת
  - קונפיגורביליות עמוקה לכללי GL מורכבים
  - ייצוא BKMV ו-PCN874 מלא
  - גרסת ענן חדשה (HCloud על AWS)
- **חולשות:** UI מיושן ותלול־הכשרה; מיועד לרו"ח ולא ליזם
- **מה לקחת:** ייצוא BKMV מלא, מודול רכוש קבוע, גמישות תרשים החשבונות

### 5.2 ריווחית (Rivhit)

- **קהל יעד:** SMB ישראלי
- **חוזקות:** עיצוב על ידי רו"ח, לוקליזציה ישראלית מלאה, ניהול מלאי משולב, סנכרון real-time בין web ומובייל, API ציבורי
- **חולשות:** מחירים לא שקופים, שיווק חלש מול מתחרים חדשים
- **מה לקחת:** קישור מלאי-לחשבונית, dashboard תזרים מזומנים יומי

### 5.3 Priority Software — ה-ERP הישראלי

- **קהל יעד:** mid-market ועד enterprise (50–1000 עובדים)
- **חוזקות:**
  - ה-ERP #1 בישראל; 15,000 לקוחות
  - GL רב-מימדי (cost centers × פרויקטים × מחלקות × אתרים)
  - אוטומציית התאמת בנק
  - תמיכה מלאה ב-PCN874, 856, 6111
  - מנוע חוקים להכרה בהכנסה
- **חולשות:** TCO גבוה, UI מורכב, דורש שותף הטמעה
- **מה לקחת:** allocation רב-מימדי לתנועות (כל פקודת יומן יודעת לאיזה פרויקט/מחלקה היא שייכת), מנוע recognition rules

### 5.4 Green Invoice (חשבונית ירוקה / Morning)

- **קהל יעד:** עצמאיים ועסקים קטנים — מובילי שוק (170K משתמשים פעילים)
- **חוזקות:**
  - חשבונית בשניות, UX חלק
  - אינטגרציית SHAAM מובנית למספרי הקצאה אוטומטיים
  - OCR לקבלות, סריקה במובייל
  - תשלום אונליין בלינק (Bit, Apple Pay, כרטיסי אשראי) → חשבונית מס/קבלה אוטומטית
  - אינטגרציות ל-Wix, WooCommerce, Shopify
  - Webhooks ו-API מתועדים
- **חולשות:** דוחות מתקדמים מוגבלים, אין GL מלא
- **מה לקחת:** UX יצירת חשבונית בקליק, לינק תשלום עצמי בחשבונית, אינטגרציית SHAAM, מובייל ראשוני

### 5.5 iCount

- **קהל יעד:** SMB עם צוות 10–100
- **חוזקות:** איחוד invoicing + CRM + מלאי + מעקב שעות; הודעת פתיחה (אימייל פתח/לחץ); תמחור לפי פרויקט

### 5.6 EZCount / SUMIT / Hashavonit

### 5.7 Cardcom

## 6. בנצ'מרק — מערכות בינלאומיות

### 6.1 Xero — מצוין בהתאמת בנק ושיתוף עם רו"ח

- **JAX (AI):** auto-reconcile של ~80% מהתנועות בבנק, לומד תבניות התאמה לפי merchant
- **Hubdoc:** OCR לקבלות שמושך נתונים ומקשר את המסמך המקורי לתנועה
- **Advisor portal:** רו"ח רואה את כל הלקוחות בלוח אחד, audit trail מלא
- **מה לקחת:** confidence threshold להתאמה אוטומטית, טבלת `transaction_rules` שלומדת מתיקוני המשתמש

### 6.2 QuickBooks Online — מצוין ב-AI לחשבוניות

- **Intuit Assist:** קח מייל מלקוח או צילום של פתק → חשבונית מוכנה
- **Bank PDF import:** ה-OCR מחלץ תנועות מ-PDF של דף בנק
- **מה לקחת:** parser שמקבל מסמך לא מובנה ומפיק שורות חשבונית; כפתור "צור חשבונית מהמייל הזה"

### 6.3 Zoho Books — מצוין באוטומציה

- **חשבוניות חוזרות + תזכורות חכמות:** זמן הלקוח (timezone), שלבי תזכורת (3 ימים לפני, ביום, 7 ימים אחרי)
- **Multi-currency:** עדכוני שערים real-time מ-Open Exchange Rates
- **מה לקחת:** סכדולר עם `pg_cron` לחשבוניות חוזרות, טבלת `invoice_reminders` נפרדת

### 6.4 Wave — מצוין בידידותיות למפתחים

- **API חינמי לחלוטין** (GraphQL + REST), webhooks לכל אירוע (`invoice.paid`, `transaction.posted`)
- **מה לקחת:** הסכמה לארכיטקטורת webhooks, GraphQL כממשק חיצוני

### 6.5 NetSuite OneWorld — מצוין בקונסולידציה רב-חברתית

- 100+ מנועי מס מקומיים, איפוס intercompany אוטומטי
- חודש סגירה מ-15 ימים ל-5
- **מה לקחת:** `company_id` כ-FK בכל טבלה, eliminations rules מוגדרים, פעם אחת בחודש אוטומציה רצה

### 6.6 Sage Intacct — מצוין ב-audit trail

- כל יצירה/עדכון/מחיקה נרשמת עם user, timestamp, old/new values
- אופציה ללוג כל **צפייה** בשדות מסומנים (HIPAA)
- AI Close Agent
- **מה לקחת:** טבלת `audit_log` immutable עם JSONB

### 6.7 FreshBooks — מצוין ב-mobile UX

- צילום קבלה במובייל → OCR → קטגוריה → תנועה (3 לחיצות)
- מעקב מיילז אוטומטי
- portal ללקוח עם תקשורת על החשבונית
- **מה לקחת:** mobile-first flow לקליטת הוצאות

### 6.8 Sage Intacct + NetSuite — דשבורדים ותחזיות

- KPI tiles עם drag-and-drop, drill-down לתנועות
- Cash flow forecast 12 חודשים, scenario analysis (אופטימי/בסיס/פסימי)
- **מה לקחת:** `MATERIALIZED VIEW` מתעדכן כל שעה לדשבורד, מודל פרופט/SARIMAX לתחזית

### 6.9 Odoo — קוד פתוח כרפרנס

- מנוע התאמה אוטומטי 95% של חשבוניות לתשלומים
- לימוד מומלץ: לקרוא את הקוד של ה-double-entry שלהם בפייתון

## 8. מצב המערכת הקיימת — מיפוי פערים

### 8.1 מה כבר יש (Apr 2026)

מבנה נתונים קיים ב-Supabase:

**`invoices`:**
`id, issued_by, sent_to, date, doc_type, invoice_num, client, before_vat, total, paid, tax_withheld, notes`

**`expenses`:**
`id, project_id, supplier, description, vat_status, amount, vat, total, paid, payment_date, has_invoice, month, notes`

**`projects`** + **`clients`** + **`artist_links`** + **`campaigns`** וכו'.

UI מקיף ב-`FinancialView.tsx` עם:
- חשבוניות + הוצאות (לפי פרויקט/אומן)

- `/api/invoices`, `/api/expenses`, `/api/projects`, `/api/clients`

### 8.3 ארכיטקטורה מומלצת

הוספת שכבת GL "מתחת" למבנה הנוכחי, בלי לשבור כלום:

```
[חשבוניות / הוצאות / שכר]   ← השכבה הקיימת (UI ידידותי)
            ↓ trigger
[journal_entries + je_lines] ← שכבת GL חדשה (double-entry)
            ↓ aggregation
[accounts (CoA) עם balance]  ← נגזר מצירוף הקווים
            ↓
[דוחות: trial balance, P&L, מאזן, מע"מ, 856, 6111]
```

כל פעולה ב-UI הקיים (יצירת חשבונית, סימון "שולם", רישום הוצאה) **תמשיך לעבוד כרגיל** — אבל ברקע תייצר אוטומטית את פקודת היומן הדו-צדדית. זה נותן לנו "הנהלת חשבונות מתחת לפני השטח" בלי לאבד את ה-UX.
